## 1. Overview

JWT Tools is a Visual Studio Code extension that gives developers a rich, interactive panel for working with JSON Web Tokens (JWTs). It is intended as a daily-use developer utility — the equivalent of jwt.io built directly into the editor.

**Core capabilities:**

| Capability | Description |
|---|---|
| **Decode** | Paste any JWT and instantly see the parsed header, payload, and signature in color-coded sections |
| **Claim Analysis** | Human-readable labels for all standard claims; timestamps shown as formatted dates with relative time ("expired 3 days ago") |
| **Verify** | Check a signature using an HMAC secret or RSA/EC public key |
| **Build** | Construct a signed JWT from a form with standard claims, custom claims, expiry shorthands, and all major algorithms |

## 3. Architecture

### 3.1 High-Level Flow

```
┌─────────────────────────────────────────────────────────┐
│                    VS Code Window                        │
│                                                          │
│  ┌─────────────────────┐    ┌──────────────────────────┐ │
│  │   Extension Host     │    │    Webview Panel (iframe) │ │
│  │   (Node.js process)  │    │    (sandboxed browser)   │ │
│  │                      │    │                          │ │
│  │  extension.ts        │◄──►│  panel.html              │ │
│  │  JwtPanel.ts         │    │  ├── HTML structure       │ │
│  │  jwtCrypto.ts        │    │  ├── CSS styles           │ │
│  │                      │    │  └── JavaScript           │ │
│  │  node:crypto         │    │      ├── UI logic         │ │
│  │  (sign/verify RS/ES) │    │      └── Web Crypto API   │ │
│  │                      │    │          (sign/verify HS*)│ │
│  └─────────────────────┘    └──────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Message Passing

Communication between the webview and extension host uses VS Code's `postMessage` / `onDidReceiveMessage` API.

**Webview → Extension Host:**

| Command | Payload | When sent |
|---|---|---|
| `sign` | `{ payload, algorithm, secret }` | Build tab, RS\*/ES\* algorithms only |
| `verify` | `{ token, secret }` | Decoder verify, RS\*/ES\* tokens only |
| `copyToClipboard` | `{ text }` | Any copy button |

**Extension Host → Webview:**

| Command | Payload | When sent |
|---|---|---|
| `prefill` | `{ token }` | User opened panel with selected text |
| `decoded` | `{ result }` | Response to `decode` command |
| `verified` | `{ result }` | Response to `verify` command |
| `signed` | `{ result }` | Response to `sign` command |
| `copySuccess` | — | Clipboard write completed |

> **Important:** HS256/384/512 signing and verification bypass the extension host entirely. They are handled in the webview using the browser's `window.crypto.subtle` (Web Crypto API). Only RS\* and ES\* algorithms require a Node.js `node:crypto` roundtrip.

### 3.3 Panel Lifecycle (Singleton)

`JwtPanel` is a singleton. At most one panel is open at any time.

```
User triggers command
        │
        ▼
JwtPanel.createOrShow(context)
        │
        ├─ Panel already exists? → reveal it (bring to front)
        │
        └─ No panel? → create WebviewPanel
                │
                ├─ Read panel.html from disk
                ├─ Inject CSP meta tag
                ├─ Set webview.html
                └─ Register message handler
```

When the panel is closed, `JwtPanel.currentPanel` is set to `undefined` and all disposables are cleaned up.

## 4. Features In Depth

### 4.1 Decoder Tab

1. User pastes a JWT into the textarea
2. An `oninput` handler fires, debounced by 150ms
3. The token is split on `.` — invalid structure shows a red error banner
4. Header and payload segments are base64url-decoded and JSON-parsed in the browser
5. The token is rendered with color-coded segments:
   - **Header** — red (`#fb015b`)
   - **Payload** — purple (`#d63aff`)
   - **Signature** — cyan (`#00b9f1`)
6. A claims analysis table enriches known claims:

| Claim | Display |
|---|---|
| `exp` | Formatted date + "Expired X hours ago" / "Expires in X days" |
| `iat` | Formatted date + relative time |
| `nbf` | Formatted date + "Not Yet Valid" badge if in the future |
| `iss`, `sub`, `aud` | Labeled with their full names |
| All others | Raw value with JSON syntax highlighting |

7. **Signature Verify:** User enters HMAC secret or PEM public key — verification runs client-side for HS\* and via extension host for RS\*/ES\*.

### 4.2 Builder Tab

1. User selects an algorithm chip (HS256, RS256, ES256, etc.)
2. Fills in standard claims form:
   - `sub`, `iss`, `aud` — free text
   - `exp` — supports shorthand: `1h`, `30m`, `7d`, `1y`, ISO date string, or Unix timestamp
   - `nbf`, `iat` — optional; `iat` defaults to current time if empty
3. Adds custom claims via dynamic table rows (type: string, number, boolean, JSON)
4. Enters signing key (HMAC secret or PEM private key)
5. Clicks **Generate Token**:
   - HS\* → signed client-side instantly via `window.crypto.subtle`
   - RS\*/ES\* → sent to extension host, signed with `node:crypto`, returned via message
6. Result shown as color-coded encoded token with **Copy** and **Send to Decoder** buttons

## 5. Design Decisions

### 5.1 Single-File Webview (`panel.html`)

All HTML, CSS, and JavaScript for the UI live in one file. This avoids a frontend build pipeline (no Webpack, no bundler) while keeping the extension lightweight (the compiled package is ~21 KB). There is no framework — vanilla JS only.

**Trade-off:** The file grows large (~65 KB source). If the UI needs significant expansion, consider splitting into separate JS/CSS files loaded via `webview.asWebviewUri`.

### 5.2 Content Security Policy

VS Code enforces a strict CSP on webview content. Without explicit configuration, all inline scripts are blocked. The extension injects this header at panel load time (in `JwtPanel._getWebviewContent()`):

```
default-src 'none'; style-src 'unsafe-inline'; script-src 'unsafe-inline';
```

`'unsafe-inline'` for scripts is necessary because the UI uses inline `onclick`, `oninput`, and `onchange` attributes throughout. This is acceptable for a local developer tool that runs only inside VS Code with no external content loaded.

**If you move event handlers to `addEventListener` calls**, you can upgrade to a nonce-based CSP:
```typescript
const nonce = crypto.randomBytes(16).toString('hex');
// CSP: script-src 'nonce-${nonce}'
// HTML: <script nonce="${nonce}">
```

### 5.3 HS\* Client-Side Signing

HMAC signing and verification use `window.crypto.subtle` (Web Crypto API available in all VS Code webviews). This eliminates the extension host roundtrip for the most common JWT algorithm family, making the builder feel instant.

### 5.4 In-Memory Keys Only

The signing key / secret is never persisted anywhere — not to VS Code `SecretStorage`, not to workspace settings, and not to disk. It lives only in the textarea's DOM value for the lifetime of that form session. This is intentional to prevent accidental secret leakage.

### 5.5 Theme Awareness

All colors use `var(--vscode-*)` CSS custom properties so the UI automatically adapts to the user's VS Code theme (light, dark, high contrast). The JWT segment colors (`#fb015b`, `#d63aff`, `#00b9f1`) are hardcoded to match the jwt.io convention and are intentionally theme-independent.

## 6. Build & Install Workflow

### 6.1 Prerequisites

| Tool | Install |
|---|---|
| Node.js 18+ | https://nodejs.org |
| npm | Bundled with Node |
| VS Code | https://code.visualstudio.com |

### 6.2 First-Time Setup

```bash
cd /Users/louiswerth/VsCodePlugins/JwtTools

# If npm cache is root-owned (one-time fix):
sudo chown -R $(id -u):$(id -g) ~/.npm

# Install dev dependencies
npm install

# Compile TypeScript
npm run compile
```

### 6.3 Development Mode (F5 Debug)

Press **F5** in VS Code while the `JwtTools` folder is open. This launches an **Extension Development Host** — a second VS Code window with the extension loaded from source. Changes to `panel.html` take effect immediately on panel reload. Changes to `.ts` files require `npm run compile` (or use `npm run watch` for automatic recompilation).

```bash
# Auto-recompile on save:
npm run watch
```

### 6.4 Package & Install

Run these three commands to rebuild and reinstall the extension permanently:

```bash
# 1. Compile TypeScript
npm run compile

# 2. Package into .vsix (runs compile again via vscode:prepublish)
npx @vscode/vsce package --no-dependencies

# 3. Install into VS Code
"/Applications/Visual Studio Code.app/Contents/Resources/app/bin/code" \
  --install-extension jwt-tools-1.0.0.vsix
```

Then reload VS Code: **Cmd+Shift+P** → `Developer: Reload Window`.

> **Tip:** Bump the `version` in `package.json` before repackaging so VS Code recognizes it as a new version and replaces the old one cleanly.

### 6.5 What Gets Packaged

The `.vsix` includes only runtime files:

```
jwt-tools-1.0.0.vsix
├── package.json
├── out/
│   ├── extension.js
│   ├── JwtPanel.js
│   └── jwtCrypto.js
└── src/webview/
    └── panel.html
```

Source `.ts` files, `node_modules/`, source maps, and dev config files are excluded by `.vscodeignore`.

## 7. Making Changes

### 7.1 Changing the UI (HTML/CSS/JS)

Edit `src/webview/panel.html`. No TypeScript compilation is needed for UI-only changes.

**To test quickly during development:**
1. Press F5 to open the Extension Development Host
2. Edit `panel.html`
3. In the Extension Development Host, close and reopen the JWT Tools panel (the HTML is re-read from disk each time the panel is created)

**To permanently install:**
```bash
npx @vscode/vsce package --no-dependencies
"/Applications/Visual Studio Code.app/Contents/Resources/app/bin/code" --install-extension jwt-tools-1.0.0.vsix
```

### 7.2 Changing JWT Decode / Sign / Verify Logic

Edit `src/jwtCrypto.ts`. This file has no VS Code dependencies — it is pure Node.js crypto.

After editing:
```bash
npm run compile
```

Exported functions:

| Function | Signature | Description |
|---|---|---|
| `decodeJwt` | `(token: string) → DecodeResult` | Base64url-decode header and payload; no verification |
| `verifyJwt` | `(token: string, secret: string) → VerifyResult` | Verify signature for HS\*/RS\*/ES\* |
| `signJwt` | `(payload, algorithm, secret, extraHeader?) → SignResult` | Create a signed JWT |

### 7.3 Changing Commands or Keybindings

Edit `package.json` under `contributes.commands` and `contributes.keybindings`.

Example — adding a new command:
```json
{
  "command": "jwt-tools.newCommand",
  "title": "JWT Tools: My New Command",
  "category": "JWT Tools"
}
```

Then register it in `src/extension.ts`:
```typescript
const cmd = vscode.commands.registerCommand('jwt-tools.newCommand', () => {
  // your logic
});
context.subscriptions.push(cmd);
```

### 7.4 Changing the Panel → Host Message Protocol

Messages are handled in `JwtPanel._handleMessage()` (`src/JwtPanel.ts`).

To add a new message type:
1. Add a `case` in the `switch` statement in `_handleMessage`
2. Call `this._panel.webview.postMessage(...)` to respond
3. In `panel.html`, add a `case` in the `window.addEventListener('message', ...)` handler
4. Add the corresponding call in the webview JS: `postMessage({ command: 'myCommand', ...data })`

### 7.5 Adding a New Standard Claim to the Builder

In `panel.html`, add a new `<div class="form-row">` block inside the **Standard Claims** card:

```html
<div class="form-row">
  <label>jti <span class="optional">jwt id</span></label>
  <input type="text" id="b-jti" placeholder="unique-token-id" />
</div>
```

Then add it to `buildPayload()` in the same file:
```javascript
const jti = document.getElementById('b-jti').value.trim();
if (jti) payload.jti = jti;
```

### 7.6 Adding a Claim Label / Enrichment

In `panel.html`, find the `CLAIM_LABELS` object and add the new claim:
```javascript
const CLAIM_LABELS = {
  // existing entries...
  my_claim: 'My Claim Label',
};
```

For time-based enrichment (date + relative time), add the claim key to the `TIME_CLAIMS` Set:
```javascript
const TIME_CLAIMS = new Set(['exp', 'iat', 'nbf', 'auth_time', 'my_time_claim']);
```

### 7.7 Bumping the Version

Edit `package.json`:
```json
"version": "1.1.0"
```

The packaged file will be named `jwt-tools-1.1.0.vsix`.

## 8. Supported Algorithms

| Algorithm | Family | Key Type | Signing Location |
|---|---|---|---|
| HS256 | HMAC | Shared secret string | Browser (Web Crypto) |
| HS384 | HMAC | Shared secret string | Browser (Web Crypto) |
| HS512 | HMAC | Shared secret string | Browser (Web Crypto) |
| RS256 | RSA | PEM private key (sign) / PEM public key (verify) | Extension Host (node:crypto) |
| RS384 | RSA | PEM private key (sign) / PEM public key (verify) | Extension Host (node:crypto) |
| RS512 | RSA | PEM private key (sign) / PEM public key (verify) | Extension Host (node:crypto) |
| ES256 | ECDSA (P-256) | PEM private key (sign) / PEM public key (verify) | Extension Host (node:crypto) |
| ES384 | ECDSA (P-384) | PEM private key (sign) / PEM public key (verify) | Extension Host (node:crypto) |
| ES512 | ECDSA (P-521) | PEM private key (sign) / PEM public key (verify) | Extension Host (node:crypto) |

## 9. Key Files Reference

### `src/extension.ts`
- Entry point called by VS Code on extension activation
- Registers two commands: `jwt-tools.openPanel` and `jwt-tools.decodeSelection`
- The decode-selection command reads the active editor's selected text and pre-fills the decoder

### `src/JwtPanel.ts`
- Manages the singleton `vscode.WebviewPanel`
- `createOrShow()` — opens or reveals the panel; accepts an optional pre-fill token
- `_getWebviewContent()` — reads `panel.html` from disk, injects the CSP meta tag, and returns the HTML string
- `_handleMessage()` — routes messages from the webview to the appropriate action (sign/verify/clipboard)

### `src/jwtCrypto.ts`
- Pure Node.js module with no VS Code dependencies
- Uses `node:crypto` built-in — zero external npm dependencies
- Handles base64url encoding/decoding, HMAC, RSA, and ECDSA operations

### `src/webview/panel.html`
- The entire visual interface in a single file (~65 KB)
- Two tabs rendered as hidden/shown `div` elements (no page navigation)
- All JavaScript is inline; state is held in module-level `let` variables
- Key sections:
  - **Lines 1–60:** CSS reset, layout, color tokens
  - **Lines 61–895:** HTML markup (tabs, cards, forms, tables)
  - **Lines 896–end:** JavaScript (VS Code bridge, utilities, Web Crypto, decoder logic, builder logic)

### `package.json`
- Extension manifest consumed by VS Code
- `contributes.commands` — palette entries
- `contributes.keybindings` — Cmd+Shift+J shortcut
- `contributes.menus.editor/context` — right-click context menu entry
- `activationEvents` — the extension only loads when a JWT Tools command is run (lazy activation)

### `.vscodeignore`
- Controls what files are included in the `.vsix` package
- TypeScript source files and `node_modules` are excluded; only compiled JS and `panel.html` are shipped

## 10. Troubleshooting

### Panel shows "Error: panel.html not found"
The extension cannot locate `src/webview/panel.html` relative to the extension path.
- **In development (F5):** Ensure the workspace root is `JwtTools/`, not a parent folder
- **After install:** Verify `panel.html` is listed in the packaged files when running `vsce package` (check the logged file list)

### Token generates but UI does not update
Most likely cause: Content Security Policy is blocking inline scripts.
- Open **Developer Tools** in the webview: `Help → Toggle Developer Tools`, then check the Console tab for CSP errors
- Ensure `JwtPanel._getWebviewContent()` is injecting the `<meta>` CSP tag before the `<head>` close

### RS256/ES256 signing fails with "Signing error"
The PEM private key format must match the algorithm:
- RS\* → `-----BEGIN RSA PRIVATE KEY-----` or `-----BEGIN PRIVATE KEY-----` (PKCS#8)
- ES\* → `-----BEGIN EC PRIVATE KEY-----` or `-----BEGIN PRIVATE KEY-----` (PKCS#8)
- Keys must be pasted with line breaks intact — some tools strip newlines

### `npm install` fails with `EACCES`
The npm cache directory has root-owned files from a previous sudo operation.
```bash
sudo chown -R $(id -u):$(id -g) ~/.npm
```
Or use the temp cache workaround: `npm install --cache /tmp/npm-cache`

### Keybinding Cmd+Shift+J conflicts with another extension
Change the keybinding in `package.json` under `contributes.keybindings`, repackage, and reinstall. You can also override it at the user level in VS Code's Keyboard Shortcuts editor without touching the source.

### Changes to `panel.html` not reflected after reinstall
VS Code caches the webview content. After reinstalling:
1. Close the JWT Tools panel (click the × on the tab)
2. Reload VS Code: `Cmd+Shift+P` → `Developer: Reload Window`
3. Reopen the panel with `Cmd+Shift+J`
